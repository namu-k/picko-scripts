# 콘텐츠 파이프라인 개선 논의

> 브랜치: `improve/content-pipeline-ux`  
> 목적: 생산 과정·결과물 관찰을 바탕으로 한 개선 방향 정리 및 논의  
> 상세: `docs/개선_논의_콘텐츠_파이프라인.md` (동일 내용, docs는 .gitignore 대상)

---

## 1. 현황 요약

- **수집**: RSS → 전처리·점수 → 다이제스트·입력 노트
- **생산**: 승인된 입력 하나당 **동시에** 롱폼 + 채널별 팩(twitter, linkedin, newsletter 등) + 이미지 프롬프트 생성
- **프롬프트**: 롱폼/팩/이미지용 지시문이 모두 `scripts/generate_content.py` 안에 하드코딩됨
- **팩 소스**: 현재는 **입력(제목·요약)** 기준으로 팩을 만들고, 메타만 `source_longform_id`로 연결 (실제 본문은 롱폼 미사용)

---

## 2. 제기된 이슈와 개선 방향

### 2.1 주제 탐색·사고 확장 단계 부재

**관찰**: 점수 높은 글 하나만 롱폼으로 확장되고, 주제가 따로 놀며 한정된 주제를 벗어나지 못함.

**제안**:
- 승인한 글을 바탕으로 **주제 탐색·사고 확장·인사이트 도출** 단계를 도입한다.
- 예: "이 주제를 더 넓히면?", "관련 논의/반론은?", "독자에게 줄 인사이트는?" 같은 메타 단계를 **별도 산출물**(예: `exploration_*.md` 또는 입력 노트의 새 섹션)로 남기고, 그 결과를 롱폼 작성 시 컨텍스트로 넘긴다.
- 구현 옵션: `daily_collector` 후처리, 또는 `generate_content` 실행 전에 돌리는 **탐색 전용 스크립트** (예: `scripts/explore_topic.py`).

**논의 포인트**: 탐색 결과를 다이제스트/입력에 어떻게 붙일지, 롱폼 프롬프트에 어떻게 주입할지.

---

### 2.2 롱폼: AI 티 나는 문체 / 프롬프트 관리

**관찰**: 글에서 AI 작성 느낌이 강함. 작성 단계의 프롬프트를 체계적으로 관리할 필요가 있음.

**제안 A – 프롬프트 관리**  
- 롱폼용 지시문을 **코드 밖**으로 분리해 버전·실험을 쉽게 한다.
  - 예: `config/prompts/longform/` 또는 `prompts/longform.md` + Jinja2 변수.
- `generate_content`는 "프롬프트 이름/경로 + 컨텍스트 변수"만 넘기고, 실제 문장은 설정/파일에서 로드.

**제안 B – 레퍼런스 기반 문체**  
- "잘 쓰인 글" 샘플을 레퍼런스로 두고, **그와 유사한 톤·구조를 내는 프롬프트**를 준비한다.
  - 예: 레퍼런스 요약 + "위 글의 톤과 구조를 참고하여 아래 주제로 글을 써주세요" 형태.
- 레퍼런스는 vault 내 전용 폴더(예: `Assets/References/`) 또는 설정에서 경로만 지정.

**논의 포인트**: 기본 프롬프트 세트 1개 vs 계정/채널별 여러 세트, 레퍼런스 선정 기준.

---

### 2.3 롱폼 → 파생 콘텐츠 생성도 승인 단계로

**관찰**: 롱폼을 만든 뒤, 그 롱폼을 바탕으로 다른 콘텐츠(팩, 이미지 등)를 만들지 여부를 사용자가 골라야 함.

**제안**:
- **1단계**: 입력 승인 → 롱폼만 생성 (또는 탐색 → 롱폼).
- **2단계**: 롱폼 검토 후 "이 롱폼으로 팩/이미지 생성"을 **별도 승인**한다.
- 데이터 모델: 입력 노트에 `writing_status` 외에, 예를 들어 `derivative_approved: true/false` 또는 롱폼 노트에 `packs_approved`, `images_approved` 같은 플래그를 두고, `generate_content`가 "롱폼까지 완료된 항목 중 파생 승인된 것만" 팩/이미지를 생성하도록 한다.
- 흐름 변경: 팩/이미지 생성 시 **소스를 입력이 아니라 롱폼 본문**으로 바꾼다 (이미 제안 2.1·2.2와 맞물림).

**논의 포인트**: 승인 UI는 다이제스트 체크리스트 확장 vs 롱폼 노트 frontmatter vs 별도 "파생 승인" 목록.

---

### 2.4 소셜(팩) 프롬프트 관리·레퍼런스

**관찰**: 소셜 미디어 포스트도 프롬프트를 관리·준비해야 함.

**제안**:
- 롱폼과 동일하게 **채널별 프롬프트를 파일/설정으로 분리** (예: `config/prompts/packs/twitter.md`, `linkedin.md`).
- 계정 프로필(`config/accounts/*.yml`)의 `channels.*.tone`, `max_length` 등과 조합해, "채널 + 톤 + 제약"이 한눈에 보이도록 한다.
- **레퍼런스**: "이 채널에서 잘 먹힌 포스트" 샘플을 두고, "이 레퍼런스의 톤·길이·구조를 참고해 아래 롱폼을 요약한 포스트를 써주세요" 형태로 확장.

---

### 2.5 소셜: 이미지·레이아웃 추천

**관찰**: 각 소셜 채널에 어떤 이미지가 들어가면 좋을지, 레이아웃까지 추천해 주면 좋겠음. 레퍼런스가 있으면 더 좋음.

**제안**:
- 채널별로 "권장 이미지 유형·비율·레이아웃"을 **설정 또는 프롬프트**로 정의한다.
  - 예: `twitter: 16:9 카드 이미지`, `linkedin: 1:1 또는 링크 미리보기`, `newsletter: 상단 히어로 + 인라인` 등.
- 생성 단계에서 "이 롱폼/팩에 어울리는 이미지 프롬프트 + 채널별 레이아웃 추천"을 한 번에 출력하거나, 기존 이미지 프롬프트 생성기에 "채널별 레이아웃 힌트" 필드를 추가한다.
- 레퍼런스: "이런 레이아웃/이미지가 잘 맞는 예시"를 vault 또는 설정에 두고, 프롬프트에서 참조하도록 한다.

**논의 포인트**: 레이아웃 추천을 "텍스트 설명만" 할지, 나중에 이미지 자동 생성까지 엮을지.

---

### 2.6 "모든 소셜로 확장" 필수 여부

**관찰**: 꼭 모든 소셜 미디어로 확장해야 할까?

**제안**:
- **채널 선택을 옵션화**: 계정에 정의된 채널 전부가 아니라, "이 롱폼에 대해 실제로 생성할 채널"을 사용자가 선택한다.
- 예: 다이제스트 또는 롱폼 단계에서 "twitter ✓, linkedin ✓, newsletter ✗"처럼 체크.
- 설정: `channels`는 "가능한 채널 목록", 실제 생성은 "이번에 선택된 채널만"으로 제한.

**논의 포인트**: 기본값은 "전부 생성" vs "전부 비선택 후 수동 선택", 다이제스트 vs 롱폼 노트에서 고를지.

## 4. 제안 작업 순서 (논의용)

1. **프롬프트 외부화** (2.2, 2.4): 롱폼/팩/이미지 프롬프트를 `config/prompts/` 등으로 분리하고, `generate_content`에서 로드하도록 변경. (공통 기반)
2. **주제 탐색 단계** (2.1): 탐색 스크립트 또는 수집 파이프라인 확장으로 "탐색 노트" 생성 후, 롱폼 입력 컨텍스트로 연결.
3. **롱폼 → 파생 승인** (2.3): 롱폼 완성 후 "팩/이미지 생성 여부" 승인 필드 도입, 팩/이미지 생성 시 소스를 롱폼 본문으로 전환.
4. **레퍼런스 기반 문체** (2.2 B, 2.4): 롱폼·팩용 레퍼런스 폴더 구조와 "레퍼런스 참조" 프롬프트 추가.
5. **채널별 이미지·레이아웃** (2.5): 채널별 이미지/레이아웃 가이드와 프롬프트 확장.
6. **채널 선택 옵션** (2.6): "이번에 생성할 채널" 선택 UI/필드 및 생성 로직에 반영.

이 순서는 의존성(프롬프트 관리가 먼저면 나머지 실험도 쉬움)과 논의 결과에 따라 바꿀 수 있음.
