# Picko 실전 운영 준비 가이드

실제로 Picko 파이프라인을 운영하기 전에 준비해야 할 정보와 단계를 정리한 문서입니다.  
**준비 순서대로** 진행하면 됩니다.

---

## 1. 환경 준비 (필수)

### 1.1 Python 및 가상환경

| 항목 | 준비 내용 |
|------|-----------|
| Python | 3.13 이상 설치 확인: `python --version` |
| 가상환경 | `python -m venv .venv` 후 활성화 (Windows: `.venv\Scripts\activate`) |
| 의존성 | `pip install -r requirements.txt` |

### 1.2 API 키 (.env)

프로젝트 루트에 `.env` 파일을 만들고, **실제 사용할 LLM에 맞는 키만** 넣으면 됩니다.

| 용도 | 환경변수 | 언제 필요? |
|------|----------|------------|
| 글쓰기(기본) | `RELAY_API_KEY` 또는 `OPENAI_API_KEY` | **필수** – 롱폼/팩 생성 시 |
| 요약·태깅 폴백 | 위와 동일 또는 `OPENROUTER_API_KEY` | Ollama 실패 시에만 사용 |
| OpenRouter 사용 시 | `OPENROUTER_API_KEY` | config에서 writer_llm provider가 openrouter일 때 |
| Anthropic 사용 시 | `ANTHROPIC_API_KEY` | writer_llm provider가 anthropic일 때 |

**준비할 정보:**  
사용할 서비스(OpenAI / Relay / OpenRouter / Anthropic) 중 하나 이상의 **API 키 값**.

- `.env.example`을 복사해 `.env`로 만들고, 해당 키만 채우면 됩니다.
- 키는 코드/문서에 붙여넣지 말고 `.env`에만 두세요.

## 2. Vault(콘텐츠 저장소) 준비

### 2.1 Vault 루트 경로

`config/config.yml`의 `vault.root`를 **실제로 사용할 폴더**로 설정합니다.

```yaml
vault:
  root: "c:/picko-scripts/mock_vault"   # 예: "D:/Obsidian/MyContentVault"
```

- 기존 Obsidian Vault 경로를 쓰거나, 전용 폴더를 새로 만들어도 됩니다.
- 스크립트가 `Inbox/Inputs`, `Content/Longform`, `Content/Packs` 등 하위 경로를 자동으로 사용합니다.

**준비할 정보:**  
콘텐츠를 저장할 **절대 경로 한 개** (예: Obsidian Vault 경로).

### 2.2 계정 정체성 파일 (롱폼/팩 품질에 중요)

글쓰기 시 **누가, 누구를 위해, 어떤 톤으로** 쓸지 정하는 마크다운 파일입니다.

| 항목 | 설명 |
|------|------|
| 기본 경로 | `{vault.root}/config/Folders_to_operate_social-media_copied_from_Vault/aa. 소셜빌더스/빌더스소셜클럽 — 계정 정체성.md` |
| 형식 | 마크다운. 문장에 `^one_liner` 같은 애노테이션으로 한 줄 소개, 타깃, 톤 등 추출 |

**준비할 정보 (문서에 넣을 내용):**

- **한 문장 소개**: 계정/채널을 한 줄로 설명하는 문장 (예: "예비·초기 창업자를 위한 실전 창업 인사이트")  
  → 해당 줄 끝에 `^one_liner` 표기
- **타깃 독자**: 누구를 위한 콘텐츠인지 (예: 예비 창업자, 초기 스타트업)
- **가치 제안**: 이 계정이 주는 핵심 가치 한 줄
- **콘텐츠 기둥(Pillar)**: P1~P4 등 주제 축 3~4개 (예: 창업 노하우, PMF, 고객 개발, 팀 빌딩)
- **톤·보이스**: 친근함/전문성/캐주얼 등
- **하지 않을 것**: 다루지 않을 주제나 스타일

경로/파일명을 바꾸고 싶다면, 정체성을 로드하는 코드에서 `relative_path`를 지정할 수 있습니다 (또는 동일 구조로 새 경로에 파일 생성).

### 2.3 주간 슬롯 파일 (선택, 권장)

해당 주에 **어떤 요일에 어떤 주제(Pillar)**로 쓸지 정하는 프리셋입니다.

| 항목 | 설명 |
|------|------|
| 기본 경로 | `{vault.root}/config/.../빌더스소셜클럽 — 주간 7개 주제 슬롯 프리셋(2-2-2-1).md` |
| 형식 | 마크다운. 요일(1–7), Pillar(P1~P4), 주제, 메모 등 |

**준비할 정보:**

- **요일별 Pillar 배분**: 예) 월=P1, 화=P2, 수=P2, 목=P3, 금=P3, 토=P4, 일=P1
- **그 주의 목표**: 고객에게 주고 싶은 한 줄 결과, 운영자 KPI, CTA 등 (있으면 좋음)

없으면 주간 슬롯 없이도 생성은 되지만, “이번 주는 이 주제” 같은 제어가 덜해질 수 있습니다.

## 3. 콘텐츠 소스 (RSS/크롤링)

### 3.1 config/sources.yml

수집할 **RSS 피드** 또는 크롤링 대상을 여기 정의합니다.

**소스 하나당 필요한 정보:**

| 필드 | 설명 | 예시 |
|------|------|------|
| id | 고유 ID (영문) | `techcrunch`, `ai_news` |
| type | `rss` 또는 `crawler` | `rss` |
| url | 피드 URL | `https://techcrunch.com/feed/` |
| category | 카테고리 (점수·필터용) | `tech_news`, `ai` |
| enabled | 사용 여부 | `true` / `false` |

**준비할 정보:**

- 수집하고 싶은 **RSS 피드 URL 목록**
- 각 소스를 구분할 **카테고리 이름** (예: tech_news, ai, newsletter)
- 카테고리별로 **일일 최대 수집 개수** (`categories.*.max_items_per_day`) – 없으면 기본값 사용

이미 예시 소스(techcrunch, hacker_news, ai_news 등)가 있으므로, URL만 바꾸거나 항목을 추가하면 됩니다.

## 4. 계정 프로필 (config/accounts/*.yml)

채널/계정별 **관심 주제, 키워드, 채널 설정**을 정의합니다.  
파일명: `config/accounts/{account_id}.yml` (예: `socialbuilders.yml`).

**준비할 정보:**

| 섹션 | 준비할 내용 |
|------|-------------|
| account_id, name, description | 계정 ID, 표시 이름, 한 줄 설명 |
| style_name | 참고할 스타일 폴더명 (`config/reference_styles/` 아래) – 없으면 `default` 등 |
| interests.primary / secondary | 관심 주제 (관련도 점수에 반영) |
| keywords.high/medium/low_relevance | 키워드 목록 (선택) |
| trusted_sources | 품질 점수에 반영할 신뢰 소스 이름 목록 (선택) |
| channels | twitter / linkedin / newsletter 등 채널별 enabled, max_length, tone, hashtags |
| content_settings | use_exploration, apply_reference_style, generate_packs, generate_image_prompts 등 |

`config/accounts/socialbuilders.yml`을 템플릿으로 복사해 계정 ID만 바꾸고, 위 항목을 채우면 됩니다.

## 5. LLM 설정 (config/config.yml)

### 5.1 어떤 조합으로 쓸지 결정

| 용도 | 기본 추천 | 준비할 정보 |
|------|-----------|-------------|
| **요약·태깅** (summary_llm) | Ollama 로컬 → 실패 시 클라우드 폴백 | Ollama 설치 여부, 사용할 모델명 (예: qwen2.5:3b) |
| **글쓰기** (writer_llm) | 클라우드 1곳 (OpenAI/Relay/OpenRouter/Anthropic) | 해당 서비스 API 키 + 모델명 (예: gpt-4o-mini) |
| **임베딩** (embedding) | Ollama 또는 local(sentence-transformers) | Ollama 임베딩 모델명 또는 local 사용 여부 |

**준비할 정보:**

- **Ollama 사용 여부**: 사용 시 `ollama pull <모델명>`으로 미리 받아두기
- **글쓰기용 서비스**: OpenAI / Relay / OpenRouter / Anthropic 중 하나 + 모델 이름
- **API 키**: 1.2의 `.env`에 넣은 키와 `config.yml`의 `api_key_env`가 일치하는지 확인

로컬만 쓰고 싶다면 summary_llm·embedding을 ollama/local로 두고, writer_llm만 클라우드로 두는 구성이 비용·품질 균형에 좋습니다.

이미 `config/reference_styles/founder_tech_brief/`가 있으면, 계정의 `style_name`만 맞추면 됩니다.

**새 스타일을 만들 때:**

- `style_extractor.py`로 참고 URL에서 스타일을 추출해 프로필 생성
- **준비할 정보**: 스타일을 따르고 싶은 **참고 글 URL 몇 개** (동일 톤/분야)

## 7. 소스와 레퍼런스 선택 가이드

### 7.1 소스(RSS 등) 고르는 구체적인 방법

수집 소스는 `config/sources.yml`에 정의합니다. **어떤 소스를 넣을지** 정할 때 아래 순서를 추천합니다.

1. **목표 정하기**  
   - “어떤 주제/분야의 글을 쓸 계정인가?” (예: 창업, AI, 마케팅)  
   - “어디에 퍼블리시할까?” (뉴스레터, 트위터, 링크드인 등)

2. **RSS URL 찾기**  
   - 좋아하는 블로그/뉴스 사이트에 들어가서 **RSS 링크**를 찾습니다.  
     - 주소창에 `/feed`, `/rss`, `/atom.xml` 등을 붙여보거나,  
     - 페이지 소스에서 `application/rss+xml`, `application/atom+xml` 링크를 검색합니다.  
   - **RSS 검색 사이트** 활용: Feedspot, RSS Search, “사이트명 + rss” 검색.  
   - **카테고리별로** 2~5개씩 골라두면 다이제스트가 균형 있게 채워집니다.

3. **카테고리 나누기**  
   - `sources.yml`의 `category`는 **점수·필터**에 쓰입니다.  
   - 계정의 `interests` / `keywords`와 맞는 카테고리 이름을 쓰면 관련도 점수가 잘 붙습니다.  
   - 예: `tech_news`, `ai`, `startup`, `marketing` 등 짧은 영문 ID.

4. **일일 상한 정하기**  
   - `categories.{카테고리}.max_items_per_day`로 **소스당/카테고리당** 하루 최대 수집 개수를 제한합니다.  
   - 너무 많으면 다이제스트가 부담스러우니, 처음엔 10~25 정도로 두고 조정하는 것을 권장합니다.

5. **테스트**  
   - `python -m scripts.daily_collector --date 오늘 --sources 소스ID --dry-run`  
   - 특정 소스만 켜서 피드가 잘 오는지, 제목/링크가 파싱되는지 확인합니다.

**정리:**  
- 소스 = “어디서 가져올지” (RSS URL + id, category, enabled).  
- 레퍼런스 = “어떤 문체/스타일로 쓸지” (아래 7.2 참고).

---

### 7.2 레퍼런스(스타일) 고르는 구체적인 방법

여기서 **레퍼런스**는 “이 사람/이 매체처럼 쓰고 싶다”는 **작성 스타일**을 의미합니다.  
스타일은 `config/reference_styles/{스타일이름}/` 폴더에 프로필로 저장되고, 계정 설정에서 `style_name`으로 지정합니다.

1. **스타일 참고할 대상 정하기**  
   - **매체/블로그**: TechCrunch, First Round Review, 특정 뉴스레터 등.  
   - **특정 글 몇 편**: 톤·구조가 마음에 드는 글 URL 3~5개.  
   - 한 계정에는 **하나의 스타일**을 붙이는 것이 기본입니다 (같은 톤 유지).

2. **스타일 프로필 만들기 (style_extractor 사용)**  
   - URL을 알고 있으면:  
     `python -m scripts.style_extractor --urls "URL1" "URL2" "URL3" --name "내스타일이름"`  
   - URL 목록이 파일에 있으면:  
     `python -m scripts.style_extractor --file urls.txt --name "내스타일이름"`  
   - 이렇게 하면 `config/reference_styles/내스타일이름/` 아래에 `profile.yml`, `writing_prompt.md` 등이 생성됩니다.

3. **계정에 연결하기**  
   - `config/accounts/{account_id}.yml`에서 `style_name: "내스타일이름"` 으로 지정.  
   - 해당 계정으로 생성되는 롱폼/팩에는 이 스타일이 적용됩니다.

4. **기존 스타일 쓰기**  
   - 이미 `config/reference_styles/founder_tech_brief/` 같은 폴더가 있다면,  
     계정 yml에 `style_name: founder_tech_brief` 만 넣으면 됩니다.  
   - 새 스타일을 만들 필요 없이 이름만 맞추면 됩니다.

**정리:**  
- 레퍼런스 = “어떤 문체로 쓸지” → `reference_styles/{이름}` + 계정의 `style_name`.  
- 소스와는 별개: 소스는 “수집할 RSS”, 레퍼런스는 “쓰기 스타일”입니다.

---

### 7.3 레퍼런스를 계정마다 / 글마다 지정할 수 있나요?

| 단위 | 지정 가능 여부 | 방법 |
|------|----------------|------|
| **계정마다** | ✅ **가능** | `config/accounts/{account_id}.yml`에 `style_name: "스타일이름"` 지정. 해당 계정으로 생성되는 모든 글에 같은 스타일이 적용됩니다. |
| **글마다** | ❌ **현재 불가** | 입력 노트(다이제스트 항목) frontmatter에 스타일/레퍼런스를 넣는 필드는 없고, `generate_content`에서도 “이 글만 다른 스타일”을 받지 않습니다. |

즉, **지금은 “계정당 하나의 레퍼런스(스타일)”만** 지원합니다.  
특정 글만 다른 스타일을 쓰고 싶다면, 현재 버전에서는  
- 그 글만 **다른 계정**으로 생성하도록 digest에서 Account를 바꾸거나,  
- 수동으로 작성하는 방식으로 우회할 수 있습니다.

**나중에 “글마다 레퍼런스”를 넣고 싶다면** (구현 시 참고용):  
- 입력 노트 frontmatter에 예: `reference_style: "다른스타일"` 필드를 추가하고,  
- `generate_content`에서 이 값을 읽어 `get_effective_prompt(..., style_name=item["reference_style"])` 처럼 넘기면,  
  계정 기본 스타일 대신 **그 글만 지정한 스타일**을 쓸 수 있게 확장할 수 있습니다.

## 8. 실전 운영 플로우 (매일/매주)

준비가 끝났다면 아래 순서로 운영할 수 있습니다.

1. **수집**  
   `python -m scripts.daily_collector --date YYYY-MM-DD`  
   (특정 소스만: `--sources techcrunch ai_news`)  
   처음엔 `--dry-run`으로 한 번 돌려보기 권장.

2. **큐레이션**  
   Vault의 `Inbox/Inputs/_digests/` 아래 해당 날짜 다이제스트를 연다.  
   생성할 항목에 체크하고, frontmatter에서 `writing_status: auto_ready`로 바꾼다.

3. **생성**  
   `python -m scripts.generate_content --date YYYY-MM-DD`  
   (전부 생성: `--auto-all`, 특정 타입만: `--type longform packs`)

4. **검증**  
   `python -m scripts.validate_output --path Content/Longform/`  
   (재귀 검증: `--recursive --verbose`)

5. **상태 점검**  
   `python -m scripts.health_check`  
   (API 키, Vault, RSS, 디스크 등)

## 9. 체크리스트 (운영 전 최종 확인)

아래를 채우면 “실전 운영 준비 완료”로 보면 됩니다.

- [ ] **환경**: Python 3.13+, venv 활성화, `pip install -r requirements.txt`
- [ ] **.env**: 글쓰기용 LLM API 키 1개 이상 설정, `config.yml`의 `api_key_env`와 일치
- [ ] **config.yml**: `vault.root`를 실제 사용할 경로로 설정
- [ ] **계정 정체성**: 정체성 마크다운 파일 존재, `^one_liner` 등 필수 문구 포함
- [ ] **주간 슬롯**: (선택) 주간 슬롯 마크다운 존재 및 형식 맞음
- [ ] **sources.yml**: 수집할 RSS URL·카테고리 반영, enabled: true
- [ ] **config/accounts/*.yml**: 운영할 계정 1개 이상, account_id·channels·interests 등 입력

## 10. 요약: “뭘 준비하면 되나요?”

| 번호 | 준비할 것 | 어디에 넣는지 |
|------|-----------|----------------|
| 1 | 사용할 LLM API 키 1개 이상 | `.env` |
| 2 | 콘텐츠 저장 경로(Obsidian Vault 등) 1개 | `config/config.yml` → vault.root |
| 3 | 계정 한 줄 소개, 타깃, 톤, Pillar | Vault 내 계정 정체성 마크다운 |
| 4 | (선택) 요일별 주제 배분 | Vault 내 주간 슬롯 마크다운 |
| 5 | 수집할 RSS URL + 카테고리 | `config/sources.yml` |
| 6 | 계정 이름, 관심 주제, 채널 설정 | `config/accounts/{account_id}.yml` |
| 7 | (선택) 참고 스타일 URL | style_extractor 또는 reference_styles 폴더 |

이 문서는 `docs/실전_운영_준비_가이드.md`에 두었습니다.  
준비하면서 경로·계정 ID·파일명이 바뀌면 이 문서의 해당 섹션만 수정해 두면 다음에 다시 운영할 때 참고하기 좋습니다.
