# 002-pipeline-ux: 파이프라인 사용성 개선

## 개요

이 브랜치는 Picko 콘텐츠 파이프라인의 **사용자 경험(UX)을 근본적으로 개선**하기 위한 기능 개발 브랜치입니다.

현재 파이프라인은 "실행하면 결과가 나온다"는 1차원적 흐름입니다. 이를 **운영자가 개입하여 품질을 조율하고, 파이프라인이 다음 단계를 안내하는** 양방향 워크플로로 발전시킵니다.

---

## 범위 (Scope)

| 구분 | 내용 |
|------|------|
| **In scope** | **(1) 품질 요구사항** — 프롬프트 개선(계정·채널·스타일 반영). **(2) 인터랙션 요구사항** — 에이전트-운영자 소통이 필요한 모든 지점에 적절한 소통 방법 및 폴백 소통 방법 마련(예: 초안 선택, 알림/완료 신호, 다음 명령 제안). |
| **Out of scope** | 발행 API 연동, 대시보드 UI, 회귀 테스트 자동화. |

이 스펙의 모든 항목은 **동등한 요구사항**이며, 유형만 아래와 같이 구분한다.

**인터랙션 원칙**: 프로그램(에이전트)-운영자 간 소통이 필요한 지점마다, **적절한 소통 방법**과 **폴백 소통 방법**이 마련되어야 한다. (예: 초안 생성 후 선택은 그런 지점의 하나이며, 알림·다음 명령 제안도 소통 방법에 해당.)

---

## 요구사항 유형

| 유형 | 설명 | 해당 항목 (002) |
|------|------|-----------------|
| **품질/생성 요구사항** | "무엇을 어떻게 만드는가" — 생성 결과의 일관성·적합성 | 영역 1: 프롬프트 개선 |
| **인터랙션 요구사항** | "에이전트-운영자 소통이 필요한 지점에 소통 방법 + 폴백 마련" | 영역 2: (예) 초안 선택, 영역 3: 알림·완료 신호, 영역 4: 다음 명령 제안 |

- **품질 요구사항**: 계정·채널·스타일이 프롬프트에 반영되어 콘텐츠 품질이 보장되는 것.
- **인터랙션 요구사항**: 소통이 필요한 지점마다 1차 소통 방법과, 그 실패/불가 시 사용할 폴백 소통 방법이 정의되는 것.

---

## 핵심 문제 정의

| 현재 상황 | 문제점 |
|-----------|--------|
| 프롬프트가 계정/채널 변수를 부분적으로만 반영 | 생성 콘텐츠의 톤·형식이 일관되지 않음 |
| 1개 초안만 생성 | 비교 선택 불가, 품질 편차 리스크 |
| 파이프라인 완료를 수동으로 확인해야 함 | 운영자가 터미널을 계속 확인해야 함 |
| 다음에 뭘 해야 할지 불명확 | "generate_content 실행했는데, 그 다음은?" |

## 4가지 개선 영역

### 영역 1: 프롬프트 개선 (P1)

**목표**: 계정 정체성, 채널 형식, 스타일 프로필이 프롬프트에 완전히 반영되도록 보장

**무엇이 바뀌나?**
```
Before: "블로그 포스트를 작성하세요"
After:  "타깃 독자(예비창업자~초기창업자)를 위해, 분석적이고 실용적인 톤으로,
         핵심 인사이트를 강조하는 블로그 포스트를 작성하세요"
```

**기대 효과**
- 계정별 고유한 톤앤보이스 유지
- 채널별 형식 요구사항 자동 반영 (트위터 280자, 링크드인 전문적 톤 등)
- 참조 URL 기반 스타일 일관성 확보

---

### 영역 2: 에이전트-운영자 소통 (P1) — 예: 다중 초안 생성 및 선택

**원칙**: 에이전트와 운영자 간에 소통이 필요한 지점마다, **적절한 소통 방법**과 **폴백 소통 방법**이 있어야 한다.

**예시(초안 선택)**: 2~3개 초안을 생성한 뒤 운영자가 선택한 초안만 최종 사용하도록 하는 경우, 소통 방법(예: Obsidian/파일에서 선택 표시)과 폴백(예: 선택 없으면 터미널 안내 또는 기본값 적용)이 정의되어야 함.

**워크플로 변화 (예시)**
```
Before: Input → [LLM] → 1개 초안 (자동 채택)
After:  Input → [LLM] → 초안 A, B, C → [소통] 운영자 선택 → 선택된 초안만 downstream 사용
        (선택 불가/미실행 시 폴백: 안내 메시지 또는 기본 초안 사용 등)
```

**저장 방식 (예시)**
- `longform_20260218.md` (초안 1)
- `longform_20260218_draft2.md` (초안 2)
- `longform_20260218_draft3.md` (초안 3)
- 프론트매터 `selected_draft: 1` 로 선택 표시

**기대 효과**
- 소통이 필요한 모든 지점에서 1차·폴백 수단이 명확함
- (예시) 품질 비교 가능 → 더 나은 콘텐츠 선택, LLM 편차 완화

---

### 영역 3: 알림 및 응답 완료 신호 (P2)

**목표**: 파이프라인 완료 시 자동 알림 + 외부 시스템 연동용 완료 신호. 소통 방법(알림 채널) + 폴백(알림 실패 시 로그/파일 등) 마련.

**지원 예정**
- 1차: 텔레그램 봇 알림 등
- 폴백: 완료 마커 파일 생성 (`pipeline_complete.marker`), 로그 기록 (알림 미설정 또는 발송 실패 시)
- 선택: 웹훅 호출

**사용 예시**
```
[텔레그램]
✅ 2026-02-18 콘텐츠 수집 완료
- 수집: 15개
- 승인 대기: 5개
- 다음: python -m scripts.generate_content --date 2026-02-18
```

**기대 효과**
- 운영자가 터미널을 지켜보지 않아도 됨
- 외부 스케줄러/봇과 연동 가능
- "완료되면 알림" → 실제로 알림

---

### 영역 4: 다음 명령 제안 (P2)

**목표**: 스크립트 종료 시 다음에 실행할 명령을 명시적으로 안내. 1차 소통은 터미널 출력; 폴백은 로그 파일 등에 동일 문구 기록.

**출력 예시**
```bash
# daily_collector 완료 후
✅ 일 수집 완료 (2026-02-18)
📋 다음 권장 명령:
   python -m scripts.generate_content --date 2026-02-18

# generate_content 완료 후 (초안 선택 필요)
✅ 콘텐츠 생성 완료
📋 다음 단계: Obsidian에서 초안을 검토하고 선택하세요
   선택 후: python -m scripts.validate_output --path Content/

# 모든 작업 완료
✅ 추가 개입 불필요. 완료.
```

**기대 효과**
- "다음에 뭘 하지?" 혼란 제거
- 워크플로의 명확한 종료점 정의
- 새 운영자 온보딩 시간 단축

## 실용적 사용 사례

### Case 1: 일일 콘텐츠 운영자

**Before**
```bash
$ python -m scripts.daily_collector
# ... 5분 대기 ...
# 완료됐나? 로그 확인...
$ python -m scripts.generate_content
# ... 생성됐나? 파일 확인...
# 이제 뭐하지?
```

**After**
```bash
$ python -m scripts.daily_collector
# ... (백그라운드 실행 후 다른 일 가능) ...

[텔레그램 알림] ✅ 수집 완료. 다음: python -m scripts.generate_content --date 2026-02-18

$ python -m scripts.generate_content --date 2026-02-18
✅ 3개 초안 생성 완료
📋 다음: Obsidian에서 초안 선택 → validate_output 실행

# Obsidian에서 초안 검토 후 선택
$ python -m scripts.validate_output --path Content/
✅ 검증 완료. 추가 개입 불필요.
```

### Case 2: 멀티 계정 운영

**시나리오**: 3개 계정(socialbuilders, tech_news, startup_kr) 운영

```yaml
# config/accounts/socialbuilders.yml
identity:
  one_liner: "예비창업자~초기창업자가 실천 가능한 인사이트를 제공"
  target_audience: "창업 0~3년, 실무 중심"
  tone_voice: "분석적, 실용적, 친근"

draft_count: 3  # 이 계정은 3개 초안 생성
```

**결과**: 각 계정별로 설정된 톤/초안 개수가 자동 반영됨

## 기술적 구현 범위

### 수정 파일

| 파일 | 변경 내용 |
|------|----------|
| `config/config.yml` | 초안 개수, 알림 설정 추가 |
| `config/prompts/*.md` | 계정/채널 변수 자리 확정 |
| `picko/prompt_composer.py` | 스타일 레이어 적용 검증 |
| `picko/notify.py` (신규) | 알림 발송 + 완료 신호 |
| `scripts/generate_content.py` | N개 초안 생성, 선택 로직 |
| `scripts/daily_collector.py` | 종료 시 다음 명령 출력 |

### 하위 호환성

- `draft_count: 1` → 기존 단일 생성 동작 유지
- 알림 미설정 → 알림 단계 스킵
- 기존 설정 파일 → 수정 없이 동작
